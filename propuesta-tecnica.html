Propuesta Tecnica - Plataforma Cliente Vecino Trebol

Este documento define la arquitectura objetivo para la solucion digital de Cliente Vecino. El backend propio gestiona identidad y datos de cliente, mientras que el sistema externo provee puntos, historial de compras y ofertas. La aplicacion frontend consume una respuesta unificada y segura.

BackendJava 21 + Spring Boot WebFlux
Base de datosPostgreSQL (clientes + credenciales)
FrontendAngular 21 (ultima major estable)
SeguridadJWT Bearer + login usuario/clave
Volver a demo Ver tarjeta movil

1. Objetivo y alcance

Objetivo

Entregar al frontend una vista consolidada del cliente por RUT en una sola consulta, combinando datos locales (cliente y autenticacion) con datos externos (puntos, compras, ofertas).

Alcance funcional

  • Autenticacion por usuario/clave y emision de token JWT.
  • Registro de nuevos clientes en base local.
  • Consulta enriquecida por RUT para puntos, compras y ofertas.
  • Manejo de fallas del sistema externo con degradacion controlada.

4. Programacion reactiva y enriquecimiento en paralelo

Se propone usar Spring WebFlux y Reactor para ejecutar en paralelo las consultas a la base local y al sistema externo. Esto reduce latencia y mantiene el flujo no bloqueante de punta a punta.

Mono<ClienteLocal> localMono = clienteRepository.findByRut(rut); Mono<ResumenExterno> externoMono = externalClient.getResumenByRut(rut) .timeout(Duration.ofSeconds(3)) .retryWhen(Retry.backoff(2, Duration.ofMillis(200))); return Mono.zip(localMono, externoMono) .map(tuple -> resumenMapper.toResponse(tuple.getT1(), tuple.getT2())) .onErrorResume(ex -> localMono.map(local -> resumenMapper.toDegradedResponse(local, ex)) );
Nota: el sistema externo es fuente de verdad para puntos, compras y ofertas; el backend solo normaliza y agrega.

6. Seguridad, resiliencia y observabilidad

Seguridad

  • Hash de contrasena con BCrypt o Argon2.
  • JWT firmado y expiracion controlada.
  • Rate limit y lockout progresivo en login.
  • No exponer credenciales ni detalles del sistema externo al frontend.

Resiliencia y operacion

  • Timeout, retry y circuit breaker en integracion externa.
  • Fallback degradado (cliente local disponible si externo falla).
  • Degradacion controlada en respuesta si el sistema externo no responde.
  • Trazabilidad con request-id, logs estructurados y metricas.

7. Plan de implementacion sugerido

Fase 1

Bootstrapping backend, modelo de datos local, login JWT y registro de clientes.

Fase 2

Integracion externa por RUT, endpoint /resumen y mapeo de contratos al frontend.

Fase 3

Hardening: observabilidad, performance, resiliencia y pruebas de carga/UAT.

Resultado esperado: experiencia rapida para usuario final y arquitectura mantenible para evolucion del producto.
